## Detection Signals

The pipeline uses **two distinct signals** that capture genuinely different properties of the text — one semantic, one structural — so combining them is more informative than either alone.

### Signal 1 — Groq LLM classifier (`llama-3.3-70b-versatile`)
- **Measures:** *semantic / stylistic coherence* — does the writing read as human or AI when judged holistically (voice, originality, idea flow, "feel")?
- **Output:** the model is prompted to return JSON `{ "verdict": "ai|human", "ai_probability": 0.0–1.0, "reasoning": "..." }`. We use **`ai_probability` — a score in [0, 1]** (probability the text is AI). Not a bare binary.

### Signal 2 — Stylometric heuristics (pure Python)
- **Measures:** *structural statistics of form* — sentence-length variance, type-token ratio (vocabulary diversity), punctuation density, average sentence complexity. No meaning, just shape and rhythm; deterministic.
- **Output:** each metric is normalized to a 0–1 "AI-likeness" sub-score, then averaged into a single **`ai_probability` in [0, 1]**. Also a score, on the same scale as Signal 1, so the two are directly comparable.

### Combining them into one confidence score
Both signals emit `ai_probability ∈ [0, 1]` on the same axis, so we fuse by **weighted average**:

```
combined_p_ai = 0.6 * llm_p_ai + 0.4 * stylometry_p_ai
```

The LLM gets the heavier weight (0.6) because it's the stronger general detector; stylometry (0.4) anchors it with an objective, gameable-but-independent structural check. Disagreement is handled *for free*: when one signal says AI and the other says human, the average is pulled toward 0.5, which automatically drives confidence down (see below). This is the mechanism that protects against confident false positives.

## Uncertainty Representation

**Confidence** is the probability of whichever class the system actually picks:

```
result      = "ai"        if combined_p_ai >= 0.70
              "human"     if combined_p_ai <= 0.30
              "uncertain" otherwise
confidence  = max(combined_p_ai, 1 - combined_p_ai)      # always in [0.5, 1.0]
```

`result` is tri-valued and mirrors the label category, so the top-level attribution is never a confident-looking "ai"/"human" when the system is actually unsure. So confidence answers "how sure are we in the call we just made?" — 0.5 is a coin flip, 1.0 is certainty.

**What a confidence of 0.6 means to the system:** the chosen class won only ~60% of the weighted evidence — a *weak* lean, barely above a guess. It sits below our high-confidence cutoff, so the system refuses to commit and surfaces the **Uncertain** label. A 0.6 is treated as "we noticed something, but don't trust this."

**Mapping raw outputs → calibrated score:** each raw signal is first normalized to a [0,1] AI-probability (the LLM returns one directly; stylometry's metrics are each scaled to 0–1 against hand-tuned human/AI reference bands, then averaged). The weighted average produces `combined_p_ai`, and confidence is derived as above. We then validate the mapping empirically (Milestone 4): run clearly-AI and clearly-human samples and confirm they land near the extremes while mixed/edited text lands mid-range.

**Thresholds — likely AI / uncertain / likely human:**

| Condition | Result | Label |
|---|---|---|
| `combined_p_ai >= 0.70` | **likely AI** | High-confidence AI |
| `combined_p_ai <= 0.30` | **likely human** | High-confidence human |
| `0.30 < combined_p_ai < 0.70` | **uncertain** | Uncertain |

Equivalently, the high-confidence labels require **confidence ≥ 0.70**; anything below is Uncertain. This guarantees a 0.51 and a 0.95 produce *different* labels, as required.

## Transparency Label Design

The exact reader-facing text for each variant. `{pct}` is the confidence rendered as a whole-number percentage (e.g. 92).

**High-confidence AI**
> **Likely AI-generated.** Our analysis indicates this content was most likely produced by an AI system (confidence: {pct}%). Two independent checks — a language-model review and a writing-structure analysis — pointed the same way. If you're the creator and believe this is wrong, you can appeal this result.

**High-confidence human**
> **Likely human-written.** Our analysis indicates this content was most likely written by a person (confidence: {pct}%). Two independent checks agreed. No action needed.

**Uncertain**
> **Inconclusive.** We couldn't determine with confidence whether this content was written by a person or generated by AI (confidence: {pct}%). Our two checks disagreed or found mixed signals, so we're not making a call — please treat this as a weak signal, not a verdict.

## Appeals Workflow

- **Who can appeal:** the content's creator (or the host platform on their behalf). In this MVP, an appeal is authorized by holding the `decision_id` returned at submission; `creator_id` is captured when provided.
- **What they provide:** the `decision_id` of the decision being contested, plus free-text `reasoning` explaining why they believe it's wrong.
- **What the system does on receipt:**
  1. Validate the `decision_id` exists (else `404`).
  2. Create an appeal record: `appeal_id`, `decision_id`, `reasoning`, `timestamp`.
  3. Update that decision's status `classified → under_review`.
  4. Append the appeal to the **audit log, next to the original decision**, so the contest lives beside what it contests.
  - No automated re-classification — the appeal flags the decision for a human.
- **What a human reviewer sees in the appeal queue:** for each `under_review` item — the original text (or reference), the original result + confidence, **both signals' individual verdicts/scores**, the exact label that was shown, the creator's reasoning, all timestamps, and current status. That's enough context to uphold or overturn without re-running anything.

## Anticipated Edge Cases

Specific scenarios this system will handle poorly, and why:

1. **Formal poetry with intentional repetition + sparse vocabulary.** A villanelle or a minimalist poem deliberately uses uniform line lengths, low type-token ratio, and repeated refrains. Stylometry reads that uniformity as machine-like and scores it AI, even though it's a human stylistic choice. *Mitigation:* the LLM signal disagrees, fusion pulls confidence down, and the result surfaces as Uncertain rather than a confident false accusation.

2. **Highly polished, formulaic human writing** (corporate press release, academic abstract, legal boilerplate). This genre is naturally uniform, hedged, and formal — exactly the texture both signals associate with AI — so they can *agree* on the wrong answer, producing a confident false positive. This is the most dangerous case because agreement raises confidence. *Mitigation:* documented limitation + the appeals path; flagged for threshold tuning in M4.

3. **Very short submissions** (a two-line micro-poem, a tweet-length excerpt). Stylometric variance and TTR are statistically meaningless on so few sentences, and the LLM has little to judge. *Mitigation:* enforce a minimum-length check at input validation and refuse to classify below it.

4. **Human-edited AI text ("AI-assisted").** Genuinely ambiguous by construction; signals tend to split. The honest output here is Uncertain — which is arguably correct, not a failure.

## Architecture

Reference diagram for Milestones 3–5. **Submission flow:** `POST /submit` runs the text through both detection signals in parallel, fuses their `ai_probability` outputs into one confidence score, generates the matching transparency label, writes the full decision to the audit log, and returns the structured response. **Appeal flow:** `POST /appeal` looks up the original decision by `decision_id`, flips its status to `under_review`, appends the appeal to the audit log beside that decision, and returns the new status.

### Flow 1 — Submission (`POST /submit`)
```
  Client
    │  raw text (JSON)
    ▼

└──────┬──────┘                  └──────────────────┘

       │ clean text
       ├───────────────────────────────┐

┌──────────────────┐           ┌────────────────────────┐

```
  Creator
    │  decision_id + reasoning (JSON)
    ▼
